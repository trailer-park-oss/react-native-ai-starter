# Prompt 4: Payments Pack — Provider Contract & Stripe Implementation

You are a senior React Native mobile payments engineer. Your task is to **implement** the payments feature pack for the CLI starter, producing real templates and pack code that integrate with the existing generator.

## Context

Prompts 1-3 are implemented. The core provides Zustand + TanStack Query. The UI pack provides design system tokens, ThemeProvider, reanimated, MMKV, and haptics. The auth pack provides React Hook Form + Zod (available as peer dependencies when auth is enabled). The payments pack (`src/packs/payments/index.ts`) is currently a stub.

### What the payments pack must do

1. Only run when `config.payments !== 'none'` (pack registry already gates this).
2. Populate `dependencies`, `expoInstallPackages`, and `ownedPaths` for Stripe.
3. Place EJS templates under `templates/payments/` that generate into the target project.
4. Run `postApplyValidation` that checks critical files exist.

## Payments Modes

- `none`: No payments SDK installed. No payments files generated. No paywall or subscription screens.

|---|---|---|

**Deduplication rule:** The pack must declare RHF + Zod in its `dependencies`/`expoInstallPackages`. The generator's `mergePackDependencies` in `src/utils/package-json.ts` handles deduplication via `Object.assign` — later packs overwrite identical keys, so this is safe.

**Why TanStack Query for payments server state:**

Products, prices, and subscription status are classic server state — fetched from a backend, cached, and invalidated on mutations. TanStack Query (already installed by core pack) is the right tool:

- `useQuery` for `getProducts()`, `getSubscriptionStatus()` — with stale-while-revalidate
- `useMutation` for `createPaymentIntent()`, `createSubscription()`, `cancelSubscription()` — with `onSuccess` cache invalidation
- Automatic retry, loading/error states, background refetch

### Payments Interface Contract

Define `src/providers/payments/payments.interface.ts`:

```ts
interface PaymentsProvider {
  initialize(config: PaymentsConfig): Promise<void>

  createPaymentIntent(params: CreatePaymentParams): Promise<PaymentIntent>
  confirmPayment(paymentIntentId: string): Promise<PaymentResult>

  getProducts(): Promise<Product[]>
  getSubscriptionStatus(): Promise<SubscriptionStatus | null>
  createSubscription(priceId: string): Promise<SubscriptionResult>
  cancelSubscription(subscriptionId: string): Promise<void>
  restorePurchases(): Promise<SubscriptionStatus | null>

  presentPaymentSheet(params: PaymentSheetParams): Promise<PaymentResult>
}
```

Define: `PaymentsConfig`, `CreatePaymentParams`, `PaymentIntent`, `PaymentResult`, `Product`, `Price`, `SubscriptionStatus`, `SubscriptionResult`, `PaymentSheetParams`, `PaymentsError`.

### Responsibility Split (Critical)

1. **Stripe SDK responsibilities (client-side):**
   - `StripeProvider` initialization with publishable key
   - Payment sheet initialization and presentation (`initPaymentSheet`, `presentPaymentSheet`)
   - Apple Pay / Google Pay readiness checks
   - Payment confirmation handling
   - Card element rendering (if needed)

2. **Backend API responsibilities (server-side, called via typed client):**
   - Create payment intent (returns `clientSecret`)
   - Create/cancel subscription
   - List products/prices
   - Query subscription status
   - Webhook handling (not generated, documented)

3. **TanStack Query layer (bridges client ↔ server):**
   - `useProducts()` — `useQuery` wrapping `getProducts()`
   - `useSubscriptionStatus()` — `useQuery` wrapping `getSubscriptionStatus()`
   - `useCreateSubscription()` — `useMutation` with cache invalidation of subscription status
   - `useCancelSubscription()` — `useMutation` with cache invalidation
   - `usePaymentSheet()` — custom hook combining intent creation + sheet presentation

### Zod Schemas for API Responses

Generate `src/providers/payments/payments.schemas.ts`:

```ts
import { z } from 'zod'

export const productSchema = z.object({
  id: z.string(),
  name: z.string(),
  description: z.string().nullable(),
  prices: z.array(z.object({
    id: z.string(),
    unitAmount: z.number(),
    currency: z.string(),
    interval: z.enum(['month', 'year']).nullable(),
  })),
})

export const subscriptionStatusSchema = z.object({
  id: z.string(),
  status: z.enum(['active', 'trialing', 'past_due', 'canceled', 'incomplete']),
  currentPeriodEnd: z.string().datetime(),
  cancelAtPeriodEnd: z.boolean(),
  product: productSchema.pick({ id: true, name: true }),
})

export const paymentIntentSchema = z.object({
  clientSecret: z.string(),
  paymentIntentId: z.string(),
  amount: z.number(),
  currency: z.string(),
})
```

Use these to validate API responses at runtime — catch backend contract mismatches early.

### Typed Backend Client

Generate `src/providers/payments/stripe/stripe-api.ts`:

```ts
import { z } from 'zod'
import { productSchema, subscriptionStatusSchema, paymentIntentSchema } from '../payments.schemas'

const API_URL = process.env.EXPO_PUBLIC_API_URL

async function apiRequest<T>(path: string, schema: z.ZodType<T>, options?: RequestInit): Promise<T> {
  const res = await fetch(`${API_URL}${path}`, {
    headers: { 'Content-Type': 'application/json', /* auth token header */ },
    ...options,
  })
  if (!res.ok) throw new PaymentsApiError(res.status, await res.text())
  return schema.parse(await res.json())
}
```

Endpoints:
- `POST /payments/create-intent` → `paymentIntentSchema`
- `POST /payments/create-subscription` → `subscriptionStatusSchema`
- `GET /payments/products` → `z.array(productSchema)`
- `GET /payments/subscription-status` → `subscriptionStatusSchema.nullable()`
- `POST /payments/cancel-subscription` → `z.object({ success: z.boolean() })`

### TanStack Query Hooks

Generate `src/providers/payments/hooks/`:

```ts
// useProducts.ts
export function useProducts() {
  return useQuery({
    queryKey: ['payments', 'products'],
    queryFn: () => stripeApi.getProducts(),
    staleTime: 5 * 60 * 1000, // products rarely change
  })
}

// useSubscriptionStatus.ts
export function useSubscriptionStatus() {
  return useQuery({
    queryKey: ['payments', 'subscription-status'],
    queryFn: () => stripeApi.getSubscriptionStatus(),
  })
}

// useCreateSubscription.ts
export function useCreateSubscription() {
  const queryClient = useQueryClient()
  return useMutation({
    mutationFn: (priceId: string) => stripeApi.createSubscription(priceId),
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ['payments', 'subscription-status'] })
    },
  })
}
```

### Error Handling

| Source Error | PaymentsError Code | User Message |
|---|---|---|
| Stripe `payment_intent.payment_failed` | `PAYMENT_FAILED` | "Payment could not be processed" |
| User cancels payment sheet | `PAYMENT_CANCELLED` | "Payment was cancelled" |
| Stripe `card_declined` | `CARD_DECLINED` | "Your card was declined. Please try another card" |
| Backend 409 subscription exists | `SUBSCRIPTION_EXISTS` | "You already have an active subscription" |
| Backend 404 no subscription | `NO_ACTIVE_SUBSCRIPTION` | "No active subscription found" |
| Network/fetch errors | `NETWORK_ERROR` | "Connection error. Check your network" |
| Stripe SDK errors | `PROVIDER_ERROR` | "Payment service error. Please try again" |
| Everything else | `UNKNOWN` | "Something went wrong. Please try again" |
